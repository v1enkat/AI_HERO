1 The AI Engine — Hybrid Architecture

The AI is a JavaScript object called AI defined in app.ai-engine.js. It contains 12+ rule-based functions plus LLM-powered wrappers that call Groq's LLaMA 3.3 70B model. The rule-based engine runs directly in the browser (offline-capable), while the LLM layer adds human-like conversational quality when connected.

// app.ai-engine.js — Hybrid AI (Rule-Based + LLM) const AI = { // Rule-based functions (work offline) chat(msg) { ... }, // keyword-based conversational AI rewriteEmail() { ... }, // pattern-based email rewriting instructorChat() { ... }, // structured 4-week course content // LLM-powered wrappers (call Groq API, fallback to above) chatLLM(msg) { ... }, // human-like chat via LLM rewriteEmailLLM() { ... }, // natural email rewriting via LLM instructorChatLLM(){ ... }, // conversational tutor via LLM callLLM() { ... }, // central Groq API caller };

3 AI Reads User Data from State (S)

The AI engine is deeply connected to the State object (S). It reads the user's tasks, moods, expenses, cycle data, energy level, sleep logs, and more to make personalized decisions:

// AI.getDashInsight() reads EVERYTHING to generate one smart sentence: getDashInsight() { const pending = S.tasks.filter(t => !t.done).length; // how many tasks? const todayMood = S.moods.find(m => m.date === today); // how is she feeling? const total = S.expenses.reduce(...); // spending status? const phase = this.getCyclePhase(); // cycle phase? // AI makes decisions based on ALL this data combined: if (todayMood === 'stressed') → "Adjusted suggestions for more breaks" if (phase === 'menstrual') → "Schedule lighter tasks, prioritize rest" if (spending > 70%) → "⚠️ Review your expenses" }

This means every response is personalized to the specific user. Two different women using the app will get completely different AI advice based on their unique data.

4 Hybrid AI — LLM with Offline Fallback

Every AI feature uses a Hybrid Architecture: it first tries the Groq LLM for human-like responses, then gracefully falls back to the built-in rule-based engine if the API is unavailable:

// AI.callLLM() — Central hybrid function async callLLM(systemPrompt, messages, fallbackFn) { const apiKey = this._getApiKey(); // config.js → Settings override if (!apiKey) return fallbackFn(); // no key? → rule-based AI try { const res = await fetch('https://api.groq.com/...', { ... }); return data.choices[0].message.content; // LLM response } catch (e) { return fallbackFn(); // network error → rule-based AI } } // Every feature uses the same pattern: chatLLM(msg) → calls callLLM() → fallback: chat(msg) rewriteEmailLLM(t,s) → calls callLLM() → fallback: rewriteEmail(t,s) instructorChatLLM(m) → calls callLLM() → fallback: instructorChat(m) generatePitchLLM(r) → calls callLLM() → fallback: generatePitch(r) socialPostLLM(t,p) → calls callLLM() → fallback: socialPost(t,p)

This means the app works in 3 modes: (1) Full LLM with API key — human-like responses, (2) No API key — rule-based AI still works, (3) Offline/API down — automatic fallback to rule-based. The app never breaks.

6 Config-Based LLM — Secure API Key Management

The Groq API key is stored in config.js — a separate file that is .gitignored (never committed to version control). The developer sets it once before deploying. A config.example.js template ships with the project.

// config.js (gitignored — developer fills this once) const CONFIG = { GROQ_API_KEY: 'gsk_your_key_here', GROQ_MODEL: 'llama-3.3-70b-versatile' }; // API key lookup chain (app.ai-engine.js) _getApiKey() { return CONFIG.GROQ_API_KEY // 1st: config.js || S.settings.apiKey // 2nd: Settings page override || ''; // 3rd: empty → use rule-based AI }
